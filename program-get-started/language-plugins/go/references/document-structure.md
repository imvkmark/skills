## 单元测试结构规范
每个知识点必须配套对应的 Go 内置 testing 单元测试，结构如下：

```go
/*
[知识点名称] 单元测试
测试目的：全面验证知识点的核心特性和用法
*/
package main

import "testing"

// Test[知识点名称] 测试[知识点名称]的基本功能
func Test[知识点名称](t *testing.T) {
	t.Parallel()

	tests := []struct {
		name string
		// 输入参数
		input interface{}
		// 期望输出
		expected interface{}
	}{
		{
			name: "测试用例1",
			input: "输入1",
			expected: "期望输出1",
		},
		{
			name: "测试用例2",
			input: "输入2",
			expected: "期望输出2",
		},
	}

	for _, tt := range tests {
		tt := tt // 捕获循环变量
		t.Run(tt.name, func(t *testing.T) {
			t.Parallel()
			// 执行测试
			result := someFunction(tt.input)
			// 验证结果
			if result != tt.expected {
				t.Errorf("expected %v, got %v", tt.expected, result)
			}
		})
	}
}

// Test[知识点名称]EdgeCases 测试边界情况
func Test[知识点名称]EdgeCases(t *testing.T) {
	t.Parallel()
	// 测试边界情况，比如空值、最大值、最小值等
}

// Test[知识点名称]ErrorCases 测试异常场景
func Test[知识点名称]ErrorCases(t *testing.T) {
	t.Parallel()
	// 测试异常场景，验证是否正确返回错误
}

// Benchmark[知识点名称] 基准测试（可选）
func Benchmark[知识点名称](b *testing.B) {
	// 基准测试代码
	for i := 0; i < b.N; i++ {
		someFunction()
	}
}
```

### 测试用例编写要求
1. **命名规范**：测试函数名要清晰表达测试意图，使用 `TestXXX` 格式，基准测试使用 `BenchmarkXXX`
2. **注释规范**：每个测试函数要有清晰的注释说明测试目的
3. **断言规范**：使用 testing 包提供的错误报告方法，错误信息要清晰
4. **测试覆盖**：必须覆盖核心功能、边界场景、异常场景
5. **独立性**：每个测试方法独立运行，不依赖其他测试的结果，使用 `t.Parallel()` 并行执行
6. **可重复性**：测试用例可以重复运行，每次结果一致
7. **表格驱动测试**：优先使用表格驱动测试模式，提高测试代码复用性

## 格式规范要求
### Markdown 格式
- 标题层级正确：使用 `#` 一级标题，`##` 二级标题，`###` 三级标题
- 代码块必须指定语言：` ```go `、` ```bash ` 等
- 列表使用统一的符号：无序列表用 `-`，有序列表用数字
- 重要内容用 `**加粗**` 标记，代码变量用 `` `反引号` `` 包裹
- 中文和英文之间要留空格，比如 `Go 1.21` 而不是 `Go1.21`

### 代码规范
- 遵循 Go 官方编码规范
- 使用 gofmt 格式化代码
- 变量名、函数名使用小驼峰命名法（首字母小写表示私有，大写表示公开）
- 类型名使用大驼峰命名法
- 常量使用大驼峰或全大写加下划线
- 注释清晰，避免无用注释
- 代码示例要完整，可以直接复制运行

### 内容规范
- 语言通俗易懂，避免过于晦涩的术语
- 示例贴近实际开发，不要用 foo/bar 这种无意义的命名
- 内容准确，避免错误或误导性的表述
- 重点突出，核心内容放在最前面
- 对于容易混淆的概念，要做对比说明
- 特别说明不同Go版本的差异，如果存在
