# 阶段模板

项目分析过程的常见阶段划分模式，支持多种架构模式和项目类型。

---

## 单体架构支持

**适用于**: 传统的单一代码库应用，如 Spring Boot、Django、Laravel 等

**第一阶段：基础结构 (2-3 篇文档)**
- 架构概览与系统目标
- 核心模块划界及目录约定
- 通用的基础类库、助手函数分析

**第二阶段：领域模块 (每个领域模块 1 篇文档)**
- 模块边界职责
- 关键服务类及对外暴露的接口
- 模块内部组件以及模块间的依赖交互

**第三阶段：数据持久化 (2 篇文档)**
- 系统全貌核心实体间的依赖网
- Repository/DAO/ORM 模式的应用方式

**第四阶段：外部接入与开放 (2 篇文档)**
- 系统提供的 API 结构
- 集成和调用的第三方系统说明

**第五阶段：跨横切面关注点 (2-3 篇文档)**
- 用户鉴权与角色权限分配机制
- 错误、日志及拦截器机制
- 系统整体性能调优设计点

**第六阶段：总结 (2 篇文档)**
- 整体评价反馈及未来重构改进意见

---

## 微服务架构支持

**适用于**: 分布式系统，多个独立服务，如 Spring Cloud、Node.js 微服务等

**第一阶段：基础设施 (2-3 篇文档)**
- 架构总览、服务分布拓扑图
- 公共库、共有契约与协议
- 配置中心及服务发现机制

**第二阶段：核心服务 (每个服务 1 篇文档)**
- 服务对外公开的接口 (API Contracts)
- 服务内部具体实现和业务代码
- 数据流分析及强依赖关系梳理

**第三阶段：网络通信 (2-3 篇文档)**
- 微服务间的通信方式（HTTP / gRPC）
- 消息队列及事件总线的订阅关系
- API 网关及聚合层逻辑

**第四阶段：数据存储层 (2-3 篇文档)**
- 各服务的独立数据库设计
- 分布式缓存策略
- 最终一致性及分布式事务保障机制

**第五阶段：监控与运维 (2-3 篇文档)**
- 日志收集、链路追踪配置
- 部署脚本、弹性伸缩方案
- 容灾与故障降级方案

**第六阶段：综合评价 (2-3 篇文档)**
- 架构合理性评估
- 提取团队代码最佳实践
- 潜在安全隐患或性能隐患反馈

---

## SDK / 库支持

**适用于**: 可复用的代码库、SDK、框架等

**第一阶段：SDK 总览 (2 篇文档)**
- SDK 对外 API 暴露边界及客户端入口
- 核心设计哲学及主要解决的痛点

**第二阶段：核心层级实现 (3-4 篇文档)**
- 梳理各种细分功能模块 (例如: 请求、签名、加密等)
- 剖析底层网络和数据流转发处理逻辑

**第三阶段：集成引导 (2 篇文档)**
- 高频调用的使用范例与坑点
- SDK 允许后续覆盖、扩展的基类或钩子

**第四阶段：工程质量 (2 篇文档)**
- 单元测试与接口测试覆盖广度
- 异常提示与文档友好度

---

## Monorepo 支持

**适用于**: 多个项目在同一仓库，如 Nx、Lerna 等

**第一阶段：仓库结构 (2-3 篇文档)**
- Monorepo 整体架构和项目划分
- 共享库和依赖管理策略
- 构建和发布流程

**第二阶段：各项目分析 (每个项目 1-2 篇文档)**
- 项目的业务目标和架构
- 项目内部的核心模块
- 项目间的依赖关系

**第三阶段：跨项目关注点 (2-3 篇文档)**
- 共享代码库的设计和使用
- 版本管理和兼容性策略
- 测试和质量保证

**第四阶段：综合评价 (2 篇文档)**
- Monorepo 架构的合理性评估
- 改进建议

## 针对不同组件提出的文档分析要素

### 通用分析要点

#### 架构与设计
- 整体架构模式和设计原则
- 主要组件及其职责
- 组件间的交互方式
- 依赖关系和耦合度

#### 代码质量
- 代码风格和命名规范
- 是否遵循 SOLID 原则
- 代码复用情况
- 技术债务识别

#### 性能与可扩展性
- 潜在的性能瓶颈
- 缓存策略
- 并发处理
- 可扩展性设计

#### 安全性
- 输入验证和防护
- 认证和授权机制
- 敏感数据处理
- 已知的安全风险

#### 测试覆盖
- 单元测试覆盖率
- 集成测试情况
- 测试质量评估

### 语言特定的分析要点

详见 `references/project-types/[language].md` 中的详细指导。

### 针对最终总结类文档的要求

**必须要包含的分析要素**:
- 归纳总结前几个阶段挖掘出的技术重点
- 基于 SOLID 原则或其他工程规范进行的架构打分
- 提取供团队持续遵守的好代码范本
- 罗列那些因为时间紧而遗留的有危害的技术负债

## 补充分析步骤参考 (常见场景)

### API 接口调用链分析

1. 观察路由定义（如 `routes/`、`@RequestMapping` 等）
2. 通过路由追踪到指定的控制器/处理器
3. 从处理器深入到业务逻辑层代码
4. 检查数据访问层的实现
5. 检视返回前的数据转换及格式化

### 定时任务 / 队列分析

1. 查看任务调度配置（时间频率、触发条件等）
2. 定位任务/队列的实现类
3. 检视任务执行的主要逻辑
4. 分析错误处理和重试机制

### 数据流分析

1. 识别数据的来源
2. 追踪数据的转换过程
3. 识别数据的存储位置
4. 分析数据的消费方

### 依赖关系分析

1. 识别直接依赖
2. 追踪间接依赖
3. 识别循环依赖
4. 评估耦合度
