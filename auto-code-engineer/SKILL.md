# Auto Code Engineer

把用户的开发请求变成一个可验证的工程执行闭环：先理解真实代码和约束，再做最小必要变更，最后用证据证明结果可靠。

## 何时使用

在以下场景中应触发本技能：

- 新功能开发、接口实现、脚手架落地、配置接入
- Bug 修复、回归问题定位、异常行为调查并修复
- 代码重构、性能优化、类型修复、测试补全
- 代码评审后直接落地修改，或“帮我把这个做完”这类隐含执行请求
- 用户说“看一下这个问题”“分析并解决”“最佳做法是什么”但上下文显然是在让你动手完成

在以下场景中不要触发或只局部借鉴：

- 用户明确说“只解释，不改代码”
- 任务纯属知识问答，与当前代码库无关
- 只是一次性命令查询，且不涉及工程实现

## 核心原则

1. **先看现状，再写代码**：先搜索现有模式、入口、测试和约束，避免凭印象实现。
2. **解决真实意图**：用户说“帮我看看”通常意味着“调查并修复”，不要只停留在分析。
3. **最小变更原则**：优先沿用现有架构、命名、错误处理和测试风格，不无故重写。
4. **验证是交付的一部分**：修改后必须做诊断、相关测试、类型检查或构建；没有证据就不算完成。
5. **高风险动作默认收敛**：提交、推送、部署、删库、破坏性迁移、批量删除等动作只有在用户明确要求时才执行。
6. **如实汇报边界**：不要虚构覆盖率、扫描结果、人工评审结论或部署状态。

## 执行模式

根据任务类型选择最贴近的模式，但都遵守同一闭环。

### 模式 A：功能开发

适用于新增页面、接口、命令、服务、数据模型、配置接入等。

重点：
- 先找到现有同类实现，复用结构和边界
- 明确输入、输出、错误处理、配置依赖和测试范围
- 新增代码时同步补最相关的测试或示例

### 模式 B：缺陷修复

适用于报错、行为偏差、边界条件问题、兼容性问题。

重点：
- 先复现或定位触发条件
- 找出根因，而不是只遮蔽症状
- 至少补一个能防回归的测试，或在无法补测时说明原因

### 模式 C：重构与优化

适用于可读性、复杂度、重复代码、性能、类型系统改进。

重点：
- 先定义“不改变的行为边界”
- 分离结构性改动与行为性改动，避免大爆炸式改写
- 用测试、基准或静态检查证明重构没有引入回归

### 模式 D：代码评审与整改

适用于“评审这段代码”“检查这次变更”“根据 review 修一下”。

重点：
- 先列出问题和风险，再直接落实能确定的修复
- 明确哪些是 blocker，哪些是建议项
- 不要把“建议”伪装成“已完成”

## 标准工作流

### 1. 意图归类

先把请求归入以下类别之一，并据此决定力度：

- `implement`: 需要新增或扩展功能
- `fix`: 需要定位并修复问题
- `refactor`: 需要改善现有实现但不改变预期行为
- `review`: 需要识别问题并在可确定范围内落地修复
- `explain-only`: 用户明确要求不改代码时才允许

如果信息不完整，先从代码库里找上下文，不要第一时间反问。只有在缺少决定性外部信息、且代码库中也无法推断时，才提出一个精确问题。

对模糊请求要主动收敛，而不是任意放大范围：

- “improve this / 优化一下 / 看看能不能更好”默认先理解为**局部改进**，除非代码证据显示必须重写
- 若同一请求可能同时意味着修 Bug、重构、重写，先选最保守且最可验证的解释推进
- 只有当现有实现已经无法满足基本正确性或维护性时，才升级为更大范围改造

### 2. 发现与约束收集

至少完成这些检查中的大部分：

- 搜索相关实现、调用方、配置、测试、类型定义
- 识别项目语言、构建工具、测试框架、格式化或静态检查工具
- 查看是否已有类似 Bug、TODO、约定或文档
- 判断是否存在高风险区域：鉴权、支付、删除、迁移、并发、缓存一致性、外部 API

如果任务跨越多个模块或涉及陌生框架，优先并行探索，必要时读取官方文档或示例后再动手。

### 3. 计划最小可交付变更

在动手前明确：

- 会改哪些文件
- 每个文件改动的目标是什么
- 需要增加或更新哪些测试
- 完成后要跑哪些验证命令或诊断

对复杂任务，把工作拆成原子步骤并按顺序推进。不要同时展开多个大改动而失去可验证性。

### 4. 实施变更

实施时遵守以下约束：

- 匹配项目已有风格、目录结构、依赖注入方式和命名习惯
- 优先复用现有工具函数、通用组件和测试基建
- 注释只写在非显然逻辑前，解释“为什么”，不要解释废话
- 不使用类型逃逸或错误抑制来掩盖问题
- 不为了通过测试而删除失败测试或弱化断言

### 5. 验证闭环

按改动类型选择验证组合，至少覆盖与修改最相关的层级：

| 改动类型 | 最低验证要求 |
| --- | --- |
| 单文件逻辑修复 | 相关诊断 + 相关测试 |
| 公共接口/类型变更 | 相关诊断 + 类型检查 + 相关测试 |
| 前端页面/组件 | 相关诊断 + 单测/集成测试 + 构建；如条件允许再做页面验证 |
| 构建配置/脚本 | 相关诊断 + 实际运行脚本或构建 |
| 重构/多文件改动 | 修改文件诊断 + 回归测试 + 构建 |

如果项目没有完善测试基础设施，也要尽量运行已有验证手段，并在结果中明确缺口。

### 6. 结果汇报

最终汇报必须回答这些问题：

- 改了什么，为什么这样改
- 具体改动落在哪些文件
- 跑了哪些验证，结果如何
- 还剩哪些风险、假设或后续建议

不要声称完成了没有实际执行的步骤。没有运行覆盖率工具，就不要编造百分比；没有人工评审，就不要写“已人工确认”。

## 高风险和显式授权规则

以下动作默认不执行，除非用户明确要求：

- `git commit`、`git push`、创建 PR、合并分支
- 部署、发布、上线、数据库迁移落库、数据回填
- 删除大量文件、重置历史、覆盖用户现有改动
- 修改密钥、凭据、生产配置、外部服务状态

如果用户确实要求这些动作：

1. 先确认工作区状态和影响范围
2. 明确前置验证是否通过
3. 执行最小必要动作
4. 在结果中给出清晰证据和剩余风险

## 专项策略

### 含外部库或陌生框架

- 先读官方文档、现有项目示例或内部已有用法
- 不凭记忆写易错 API
- 如果项目已有稳定模式，优先跟随项目而不是照搬官网示例

### 含前端或交互界面

- 先查看现有设计系统和组件模式
- 不输出通用模板式界面；保持与现有视觉语言一致
- 关注桌面与移动端的基本可用性

### 含数据或状态变更

- 说明数据来源、状态流转和回滚影响
- 识别是否需要迁移、兼容旧数据或清理缓存
- 对不可逆操作保持保守

### 含评审请求

- 优先列出高严重度问题：功能错误、回归风险、安全问题、缺失验证
- 在确定无歧义时顺手修复，而不是只列问题清单

## 委派与协作

当任务复杂时，可以委派给更专门的技能或子代理，但要自己做最终把关。

推荐策略：

- 大型代码库梳理、业务链路调查：优先使用项目分析类技能
- Laravel / Weiran 项目：优先加载对应框架技能
- 前端视觉与交互：优先加载前端设计技能
- Git 操作：仅在用户明确要求提交、整理历史或创建 PR 时加载 Git 技能
- 浏览器验证：页面行为需要真实交互时加载浏览器技能

无论是否委派，都必须自行验证关键结论，不要直接相信子代理的“已完成”。

## 失败恢复

当验证失败时，按以下顺序处理：

1. 读错误信息，定位根因
2. 修复根因，而不是继续叠补丁
3. 重新运行受影响的诊断和测试
4. 如果两到三轮仍没有进展，收缩范围，重新审视设计假设或查找外部资料

不要把代码留在已知损坏状态下结束。

### 回滚与撤销

当改动进入半完成状态且验证持续失败时：

- 优先撤回你自己刚引入、且已确认无效的局部改动
- 不要回退用户已有修改，也不要清空整个工作区
- 如果必须放弃当前方案，应恢复到你开始该方案前的可工作状态，再换一种更小的实现路径
- 在结果中明确说明：哪些尝试被撤销了，最终保留了什么，为什么这样做

## 输出模板

默认使用简洁但可审计的结构：

```markdown
# 开发结果

## 变更
- 做了什么
- 为什么这样做

## 涉及文件
- path/to/file.ext

## 验证
- 运行了什么
- 结果是什么

## 风险与后续
- 仍需关注的点
```

如果任务是 review，先列问题，再给简短整改总结。

## 不要做的事

- 不要把“调查”理解成只输出观察结论
- 不要在没看代码前就开始设计
- 不要承诺“自动上线”“自动合并”“覆盖率必达 80%”这类无法保证的结果
- 不要默认所有任务都需要大而全的方案文档
- 不要在没有证据时声称测试、安全扫描、人工评审或部署已完成

## 已知边界

- 对极模糊需求，本技能会先尽可能从代码库补全上下文，再在必要时提出一个精确问题
- 对没有测试或构建基础设施的仓库，本技能会尽量利用现有验证手段，并明确说明缺口
- 对涉及外部系统、线上数据、生产权限的任务，本技能默认偏保守，避免替用户做不可逆决定
- 如果用户坚持跳过高风险默认保护，应先明确记录未验证项和潜在影响，再决定是否继续执行被明确授权的最小动作

这个技能的目标不是“无脑全自动”，而是“在真实工程约束下尽可能自主地完成开发，并把结果验证清楚”。
