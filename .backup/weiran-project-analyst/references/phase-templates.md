# 阶段模板

项目分析过程的常见阶段划分模式，重点支持适用于 Weiran 框架的项目。


---


## Weiran 框架应用

**第一阶段：Weiran 框架基础 (2-3 篇文档)**
- 整体架构概述以及项目中体现出的 Weiran 框架特性
- 项目基础目录结构及核心配置 (config/ 目录) 作用分析
- 服务提供者 (ServiceProviders) 和依赖注入绑定分析

**第二阶段：核心应用模块 (每个功能模块 1 篇文档)**
- API/Web 控制器的请求处理链路
- 核心模型 (Models) 及它们在业务上的含义
- Action 层/服务层的业务逻辑封装
- 数据访问与持久化层

**第三阶段：Weiran 特色功能 (2-3 篇文档)**
- 事件监听系统的运用 (Events / Listeners)
- 定时任务调度机制 (Schedule / Commands)
- 异步队列排队与消费 (Jobs/Queues)
- 权限与中间件保护 (Middleware / Policies)

**第四阶段：外部接口与互操作 (2 篇文档)**
- 跨系统 API 接口安全性分析
- 第三方服务（如短信、支付等 SDK）接入实现分析

**第五阶段：数据支撑与性能层 (2-3 篇文档)**
- 核心业务表数据库设计梳理
- 系统缓存 (Redis 等) 策略及粒度
- 平台导入导出、文件上传下载功能分析

**第六阶段：系统与运维支持 (2-3 篇文档)**
- 统一日志系统的记录与分类排查策略
- 全局异常捕获及错误响应格式
- 系统关键点或服务可用性相关策略分析

**第七阶段：综合总结 (2 篇文档)**
- 整体架构设计的合理性评估
- 代码中值得推广的最佳实践
- 亟待优化或重构的技术债务建议

## 微服务架构支持

**第一阶段：基础设施 (2-3 篇文档)**
- 架构总览、服务分布拓扑图
- 公共库、共有契约与协议
- 配置中心及服务发现机制

**第二阶段：核心服务 (每个服务 1 篇文档)**
- 服务对外公开的接口 (API Contracts)
- 服务内部具体实现和业务代码
- 数据流分析及强依赖关系梳理

**第三阶段：网络通信 (2-3 篇文档)**
- 微服务间的通信方式（HTTP / gRPC）
- 消息队列及事件总线的订阅关系
- API 网关及聚合层逻辑

**第四阶段：数据存储层 (2-3 篇文档)**
- 各服务的独立数据库设计
- 分布式缓存策略
- 最终一致性及分布式事务保障机制

**第五阶段：监控与运维 (2-3 篇文档)**
- 日志收集、链路追踪配置
- 部署脚本、弹性伸缩方案
- 容灾与故障降级方案

**第六阶段：综合评价 (2-3 篇文档)**
- 架构合理性评估
- 提取团队代码最佳实践
- 潜在安全隐患或性能隐患反馈

## 针对不同组件提出的文档分析要素

### Weiran 组件特有关注点

#### 控制器分析重点
- 路由与控制器的精准映射关系及相关请求体 (FormRequests)
- 控制器中是否存在越界的重业务逻辑处理
- 返回的响应格式要求（例如是否遵循特定的标准 JSON 封装）
- 控制器层级的访问控制及中间件使用

#### 模型 (Model) 分析重点
- 关联关系是否合理定义（一对一、一对多、多对多）
- 是否使用查询构建器进行了过度的全局范围过滤 (Global Scopes)
- 模型监听事件（creating、updating、deleted）对数据的副作用

#### 服务/Action 层分析重点
- 控制器提取出来的主要业务是如何通过类方法包裹的
- 是如何与第三方 API / 数据库 进行数据交互的
- 接收参数的验证转换及异常向外抛出的格式和策略

#### 中间件与配置
- 系统路由中注入的中间件到底发挥了何种拦截
- 获取及加载自定义配置的链路

### 针对微服务的深度探索要求

**必须要包含的分析要素**:
- 确定服务接口契约（接收什么，返回什么）
- 承载的核心业务步骤
- 本地数据表定义
- 该服务又向哪些下游服务发出了远程请求
- 当下游部分失败时的重试熔断机制

### 针对数据层的深度探索要求

**必须要包含的分析要素**:
- 表结构的字段类型及关系分析
- 是否建立了合理的索引
- 是否存在 N+1 请求及对应优化策略
- 并发时使用的锁机制与事务边界
- 随时间推移的历史遗留数据清理和迁移方案

### 针对性能瓶颈的深度探索要求

**必须要包含的分析要素**:
- 定位可能引发超时阻塞的慢请求代码块
- 是否存在同步调用的远端 API，是否可以改为异步或队列
- 缓存命中率以及是否会出现缓存击穿、穿透风险
- PHP-FPM / DB 连接池等资源利用上的潜在风险

### 针对最终总结类文档的要求

**必须要包含的分析要素**:
- 归纳总结前几个阶段挖掘出的技术重点
- 基于 SOLID 原则或其他工程规范进行的架构打分
- 提取供团队持续遵守的好代码范本
- 罗列那些因为时间紧而遗留的有危害的技术负债

## 补充分析步骤参考 (常见场景)

### API 接口调用链分析
1. 观察 `routes/` 目录下（如 `api.php`, `web.php`）注册的路由。
2. 通过路由追踪到指定的控制器。
3. 从控制器的方法体深入到 `Action` 服务层代码。
4. 检查服务内部的模型数据抓取和关联加载。
5. 检视返回前的结构体封装及字段修剪。

### 定时任务 / 队列分析
1. 查看 `app/Console/Kernel.php` 中的调度记录（时间频率）。
2. 在 `Commands/` 目录中定位触发的任务类体。
3. 或者探查 `Jobs/` 目录下的队列实现，检视 `handle` 以及 `failed` 方法执行的逻辑。
