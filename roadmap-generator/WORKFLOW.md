# 详细工作流

## 模式 1：规划模式（生成学习计划）

当用户要求创建学习计划、更新计划，或输出目录下尚未存在计划文件时，进入此模式。

### 步骤 1.1：需求分析与领域建模

理解用户的学习需求和目标：

1. **确定目标编程语言**（Java/Python/Go/PHP 等）
2. **确定学习领域**（基础、核心类库、Web 开发、并发编程等）
3. **评估学习者水平**（入门、中级、高级）
4. **明确学习目标**（面试准备、项目开发、技能提升等）
5. **估算可用学习时间**，合理规划内容量
6. **默认提供完整学习路径**，需用户确认后开始执行

### 步骤 1.2：设计分阶段学习计划

创建结构化的分阶段学习计划，将知识点组织成逻辑单元。

**粒度划分**：
- **入门学习**：3 级结构（阶段 → 模块 → 知识点）
- **中高级/面试学习**：5 级结构（阶段 → 模块 → 章节 → 知识点 → 子知识点）

**时间估算**：
- 每个知识点：30-60 分钟
- 每个子知识点：10-20 分钟

**参考资源**：
- 对应语言插件中的标准阶段划分模板（见 `language-plugins/[语言]/references/phase-templates.md`）

### 步骤 1.3：生成计划文件

在输出目录中创建两个核心计划文件：

#### 文件 1：`plan-learning.md`（学习计划主体）
- 完整的知识点清单，分阶段组织
- 每个知识点的难度、预计学习时间、学习目标
- 文档编写规范和输出格式要求

#### 文件 2：`plan-progress.md`（进度追踪文件）

**Metadata 部分**：
```markdown
## Metadata
- Plan ID: [唯一生成的ID]
- Target Language: [目标编程语言]
- Learning Path: [学习路径类型]
- Learning Path Level: [full (5级) / simple (3级)]
- Version: 1
- Created: [创建时间]
- Last updated: [时间戳]
- Status: 待确认 / 待执行 / 执行中 / 已完成
- Current phase: [当前执行的阶段编号]
- Last completed task: [None]
- Auto-execution: disabled
- Parallel agents: 3
- Adjustment requests: []
- Change history: []
```

**进度追踪表格**：
- **5 级结构**：阶段 | 模块 | 章节 | 知识点 | 子知识点 | 状态 | 完成度 | 负责人 | 备注
- **3 级结构**：阶段 | 知识点 | 状态 | 完成度 | 负责人 | 备注

### 步骤 1.4：请求用户确认

完成规划后，展示计划摘要并请求确认：

```markdown
## 学习计划已生成

**目标语言**: [语言名称]
**学习路径**: [路径类型]
**知识点总数**: [数量] 个
**预计完成时间**: [X] 小时

### 计划摘要
- 第1阶段：[阶段名称] ([知识点数] 个知识点)
- 第2阶段：[阶段名称] ([知识点数] 个知识点)
- ...

### 当前状态
- 表格层级：[5级/3级]
- 版本：v[版本号]

---

**请确认是否开始执行？**
- [ ] 确认执行
- [ ] 调整部分内容（请说明需要修改的部分）
- [ ] 重新规划
```

用户确认后，将 Metadata 中的 `Status` 从「待确认」更新为「待执行」。

## 模式 2：执行模式（生成学习内容）

此模式要求输出目录下必须存在已建好的计划文件：
- `{output-dir}/plan-learning.md`
- `{output-dir}/plan-progress.md`

如果这些文件缺失，自动切换回**模式 1：规划**。

### 步骤 2.0：增量更新检测

在开始执行前，检查是否需要增量更新：

1. 扫描 `{output-dir}/docs/` 目录，获取已存在的文档列表
2. 扫描 `{output-dir}/src/test/[语言]/` 目录，获取已存在的测试文件列表
3. 在 `plan-progress.md` 中标记「待执行」的任务时：
   - 文档和测试都已存在 → 跳过该任务（标记为「已跳过」）
   - 只有文档或只有测试 → 补充缺失部分
   - 都不存在 → 正常执行
4. 向用户报告增量更新的结果

### 步骤 2.1：加载上下文与任务分发

1. 读取 `plan-learning.md` 和 `plan-progress.md`
2. 识别当前阶段、模块、章节和待执行的知识点任务
3. 加载对应语言插件的配置和规范
4. 任务粒度适配：
   - 简单知识点：单个子 agent 负责完整知识点
   - 复杂知识点：拆分多个子任务，由不同子 agent 负责
5. 根据 `parallel-agents` 配置，将任务分发给多个子 agent 并行执行
6. 确保任务之间无依赖，避免循环等待

### 步骤 2.2：子 Agent 任务执行

每个子 agent 执行以下标准化流程：

#### 2.2.1 知识点文档生成
遵循对应语言插件中的文档结构规范，生成包含以下内容的学习文档：
- 概念讲解和核心特性
- 实际使用场景说明
- 可运行的代码示例（带详细注释）
- 最佳实践和常见坑点
- 相关面试题和解答思路

#### 2.2.2 单元测试生成
生成配套的对应语言的单元测试用例：
- 覆盖核心用法、边界场景、异常情况
- 遵循对应语言的编码规范和测试框架最佳实践
- 测试用例可以直接运行，无额外依赖

#### 2.2.3 内容自校验
每个子 agent 完成内容生成后，必须进行自校验：
- [ ] 文档结构符合对应语言的规范
- [ ] 代码示例可以正常编译运行
- [ ] 测试用例全部可以通过
- [ ] 内容准确无误，没有概念错误

#### 2.2.4 代码执行验证（必须执行）
自校验完成后，必须实际运行代码验证：

**编译验证**：
- Java: `mvn compile` 或 `javac`
- Python: `python -m py_compile`
- Go: `go build`
- PHP: `php -l`（语法检查）

**测试执行**：
- Java: `mvn test`
- Python: `pytest`
- Go: `go test ./...`
- PHP: `./vendor/bin/phpunit`

**验证结果处理**：
- 编译失败 → 修复代码错误后重新验证
- 测试失败 → 修复测试或代码后重新验证
- 只有编译和测试都通过才算验证完成

### 步骤 2.3：结果汇总与质量检查

主 agent 收集所有子 agent 的输出结果，进行统一质量检查：

1. **格式检查**：所有文档和测试文件符合统一的格式规范
2. **内容检查**：知识点讲解准确，代码示例正确
3. **一致性检查**：相关知识点之间的表述一致，没有矛盾
4. **完整性检查**：所有计划中的内容都已生成，没有遗漏

如果发现内容缺陷，将任务退回给对应的子 agent 进行修正。

### 步骤 2.4：更新执行进度

每完成一个知识点的生成和校验后，立即更新 `plan-progress.md` 文件：

1. 更新进度表格中的状态和完成度
2. 更新 Metadata 中的时间戳和当前阶段
3. 如果所有任务完成，将状态更新为「已完成」

### 步骤 2.5：阶段迭代

完成当前阶段的所有任务后：

1. 向用户汇报当前阶段完成情况
2. 请求确认是否继续下一阶段
3. 获得确认后，加载下一阶段任务继续执行

## 错误处理与恢复

### 任务失败处理

当子 agent 执行任务失败时，按以下流程处理：

**失败检测**：
- 子 agent 报告执行状态为「失败」
- 代码编译/测试执行返回非零退出码
- 任务超时（默认 10 分钟）

**重试策略**（最多 2 次）：
- 第 1 次失败：记录失败原因，重新分配任务
- 第 2 次失败：更换执行 agent 或简化任务粒度
- 第 3 次失败：将任务标记为「阻塞」，记录到 `Adjustment requests`

**失败上报**：
```markdown
## 任务执行受阻

**受阻任务**: [知识点名称]
**失败原因**: [具体错误信息]
**已尝试次数**: X/3

**建议处理方式**:
- [ ] 跳过此知识点，继续其他任务
- [ ] 简化任务范围后重试
- [ ] 手动处理后继续
```

### 中断恢复

如果执行过程中断：

1. 读取当前的 `plan-progress.md` 状态
2. 从「待执行」或「进行中」的任务继续
3. 不重复执行已完成的任务

在执行过程中，展示进度：

```markdown
## 当前进度

[████████████░░░░░░░░░░░░░] 50% (5/10 个知识点)
- ✅ 已完成: 知识点A, 知识点B, 知识点C, 知识点D, 知识点E
- 🔄 进行中: 知识点F
- ⏳ 待执行: 知识点G, 知识点H, 知识点I, 知识点J
- ❌ 已失败: 0
```
