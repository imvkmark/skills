# 多 Agent 协作规范
本文档定义了多 agent 协作的任务分发、执行和结果汇总规范，确保多个子 agent 能够高效协同工作。

---

## 角色分工

### 主 Agent (协调者)
**职责**：
1. 负责整体规划和任务分发
2. 监控子 agent 的执行进度
3. 汇总和校验子 agent 的输出结果
4. 处理冲突和异常情况
5. 更新进度追踪文件
6. 与用户进行交互

**能力要求**：
- 具备全局视野和规划能力
- 能够理解和拆解复杂任务
- 具备质量评估和冲突解决能力

### 子 Agent (执行者)
**职责**：
1. 接收并执行分配的具体知识点任务
2. 按照规范生成文档和测试用例
3. 进行内容自校验
4. 及时上报执行状态和遇到的问题
5. 根据反馈修正输出内容

**能力要求**：
- 精通 Go 技术栈的特定领域
- 能够独立完成高质量的内容生成
- 具备严格的质量意识

---

## 任务分发机制

### 任务拆分原则
1. **独立性**：每个任务对应一个独立的知识点，任务之间无依赖关系，可以并行执行
2. **粒度适中**：每个任务的预计执行时间在 10-30 分钟之间
3. **边界清晰**：每个任务的输入、输出和验收标准明确
4. **负载均衡**：任务难度和工作量尽量均衡分配

### 任务分配策略
1. **按领域分配**：根据子 agent 的专长分配对应领域的任务
   - 基础语法任务 → 分配给擅长基础的 agent
   - 并发编程任务 → 分配给擅长 goroutine/channel 的 agent
   - 微服务任务 → 分配给擅长云原生开发的 agent

2. **难度匹配**：
   - 入门难度任务 → 可以分配给所有 agent
   - 中级难度任务 → 分配给有一定经验的 agent
   - 高级难度任务 → 分配给经验丰富的 agent

3. **动态调整**：
   - 如果某个 agent 执行速度快，可以分配更多任务
   - 如果某个 agent 多次出现质量问题，调整到难度较低的任务

### 任务描述格式
每个子 agent 收到的任务必须包含以下信息：
```typescript
{
  "task_id": "唯一任务ID",
  "knowledge_point": "知识点名称",
  "difficulty": "入门/中级/高级",
  "estimated_time": "预计完成时间",
  "requirements": {
    "document": true,
    "code_example": true,
    "unit_test": true,
    "specific_requirements": []
  },
  "output_spec": {
    "document_path": "docs/[知识点名称].md",
    "code_path": "[知识点名称].go",
    "test_path": "[知识点名称]_test.go"
  },
  "deadline": "截止时间"
}
```

## 执行流程规范

### 子 Agent 执行流程
1. **任务接收**：接收主 agent 分发的任务，确认任务要求
2. **需求分析**：理解知识点的核心内容和输出要求
3. **内容生成**：
   - 编写学习文档（遵循 document-structure.md 规范）
   - 编写代码示例（符合 Go 风格规范，可编译运行）
   - 编写单元测试（使用 Go 内置 testing 包）
4. **自校验**：
   - 文档结构符合规范要求
   - 内容准确，没有概念错误
   - 代码示例可以正常编译运行 (`go build`)
   - 单元测试全部通过 (`go test`)
   - 格式符合 gofmt 规范
5. **结果提交**：将生成的文件提交给主 agent，同时提交自检报告

### 状态上报
子 agent 需要及时向主 agent 上报状态：
- 任务开始执行 → 上报 "进行中" 状态
- 遇到无法解决的问题 → 上报 "阻塞" 状态，说明问题详情
- 任务完成 → 上报 "已完成" 状态，提交输出结果

## 结果汇总与质量检查

### 主 Agent 检查流程
1. **格式检查**：
   - 文件命名是否符合规范
   - 文档结构是否完整
   - 代码格式是否符合 Go 官方风格指南
   - Markdown 格式是否正确

2. **内容检查**：
   - 知识点讲解是否准确
   - 代码示例是否正确可运行
   - 测试用例是否覆盖核心场景
   - 内容是否有遗漏或错误

3. **一致性检查**：
   - 相关知识点之间的表述是否一致
   - 术语使用是否统一
   - 示例代码风格是否统一

4. **完整性检查**：
   - 所有要求的输出文件是否都已生成
   - 每个文件的内容是否完整
   - 是否符合计划中的要求

### 问题处理机制
1. **轻微问题**（格式错误、错别字等）：主 agent 可以直接修正
2. **中等问题**（内容不准确、示例有 bug 等）：退回给对应子 agent 修正
3. **严重问题**（概念错误、结构混乱等）：重新分配任务给其他 agent

## 并行执行最佳实践

### 任务调度
1. **批量分发**：一次性分发多个任务，减少调度开销
2. **优先级调度**：核心知识点、基础知识点优先分配和执行
3. **预留缓冲**：保留 1-2 个 agent 作为机动力量，处理突发任务

### 结果收集
1. **异步收集**：使用 `background_output` 工具异步收集结果
2. **批量处理**：收集到一定数量的结果后再进行批量检查
3. **超时处理**：设置任务超时时间，超过时间未完成的任务重新分配

### 冲突解决
1. **内容冲突**：如果多个 agent 对同一个知识点的表述不一致，主 agent 参考权威资料进行仲裁
2. **资源冲突**：如果多个 agent 需要修改同一个文件，主 agent 协调修改顺序
3. **进度冲突**：如果某个 agent 进度落后，主 agent 可以将其部分任务重新分配给其他 agent

## 性能指标
理想情况下的多 agent 执行效率：
- **并行度**：3 个 agent 并行执行，效率提升约 2.5 倍
- **吞吐量**：每小时可以生成 6-9 个知识点的完整内容
- **错误率**：内容错误率低于 5%
- **返工率**：需要退回修正的任务比例低于 10%

通过合理的任务拆分和调度，多 agent 协作模式可以大幅提升大规模学习内容的生成效率。
